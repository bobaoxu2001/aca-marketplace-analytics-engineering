# Metric-Grounded LLM Agents for Reliable Domain Analytics

## Abstract

Large language model analytics agents can translate natural-language questions
into plausible business conclusions, but domain analytics requires more than
fluent natural language or syntactically valid SQL. This project proposes a
metric-grounded workflow for answering ACA Marketplace analytics questions over
official CMS public data. The system constrains answers through a semantic metric
registry, dbt-modeled marts, executable SQL validation, result-row citations, and
unsupported-claim detection. We introduce a benchmark of 30 natural-language
questions spanning premiums, issuer competition, plan availability, benefit
coverage, plan continuity, and quality ratings. Results are intentionally left to
be populated by the reproducible evaluation runner once local CMS data and dbt
marts are available.

## 1. Introduction

Analytics agents are often evaluated as natural-language-to-SQL systems. In
business domains, however, a correct-looking query is not enough. Stakeholders
need to know which metric definition was used, whether the query executed,
whether numerical claims are backed by result rows, and whether the answer
overstates what the data can support.

The ACA Marketplace domain is a useful testbed because the public CMS PUFs are
large, multi-table, and semantically dense. Premiums, plan attributes, service
areas, crosswalk history, benefit cost sharing, and quality ratings each have
different grains and caveats. This makes hallucination easy: an agent may infer
competition, affordability, or quality conclusions that are not supported by the
available public data.

## 3. Dataset

The benchmark uses official CMS ACA Marketplace Plan Year 2026 Public Use Files
already supported by the parent analytics engineering project:

| Source | Approximate rows |
| --- | ---: |
| Rate PUF | 2.24M |
| Benefits and Cost Sharing PUF | 1.46M |
| Plan Attributes PUF | 22k |
| Service Area PUF | 8.8k |
| Plan ID Crosswalk PY2025 to PY2026 | 158,746 |
| Quality PUF | 4,302 |

The data is modeled through DuckDB and dbt into marts for premiums, plans,
issuers, geography, availability, benefits, plan history, and quality ratings.
All generated outputs must be regenerated from local public-data files; no
benchmark answer numbers are fabricated in the repository.

## 4. Method

We compare three variants.

### A. Direct LLM Baseline

The model receives the user question and returns an answer without database
execution. This measures fluent unsupported answering risk. The committed
adapter does not call a model unless the user configures credentials.

### B. LLM + SQL Baseline

The model receives schema context, generates SQL, executes against DuckDB, and
returns a result. The current local harness uses gold SQL as a deterministic
stand-in until a live SQL-generation adapter is configured.

### C. Metric-Grounded Agent

The metric-grounded agent maps each question to approved metric definitions,
uses constrained SQL over approved mart tables, validates the query, executes it,
and returns an answer with metric, table, and result-row citations. Unsupported
or unavailable answers are marked instead of guessed.

## 5. Benchmark

The benchmark contains 30 questions across three difficulty levels.

| Area | Example |
| --- | --- |
| Premiums | Which states have the highest median silver plan premiums? |
| Competition | Which counties have limited issuer choices? |
| Availability | Which metal levels dominate plan availability? |
| Continuity | Which states have the highest share of new current plans? |
| Quality | Are higher quality ratings associated with higher premiums? |
| Benefits | Which benefit categories differ most across metal levels? |

Each question has gold SQL under `benchmark/gold_sql/` and can generate
structured JSON gold answers under `benchmark/gold_answers/`.

## 6. Evaluation

- SQL execution correctness,
- numerical accuracy against gold answers,
- unsupported claim rate,
- citation coverage,
- traceability score,

## Reproducibility Notes

The full experimental run requires six official PY2026 public CMS files:
Rate PUF, Plan Attributes PUF, Benefits and Cost Sharing PUF, Service Area PUF,
Plan ID Crosswalk PUF, and Quality PUF. The exact official CSV and CMS ZIP URLs
are recorded in `configs/data_sources.yaml`.

Run sequence:

```bash
python3 scripts/download_cms_pufs.py
python3 scripts/load_to_duckdb.py
cd dbt_project && dbt build --profiles-dir . && cd ..
python3 research/metric_grounded_llm_agents/benchmark/generate_gold_answers.py
python3 research/metric_grounded_llm_agents/evaluation/run_eval.py
```

Gold answers and evaluation results should be reported only after this sequence
completes against the real local DuckDB/dbt marts.

## 7. Limitations

This project does not claim causal relationships between quality, premiums,
competition, and plan design. It uses public CMS files and does not include
enrollment weights, claims, subsidies, consumer choice behavior, or private
issuer data. The current live LLM adapters are placeholders unless credentials
are configured. The metric mapper is benchmark-driven and should be extended for
unseen free-form questions.

## 8. Conclusion

Metric grounding reframes analytics agents as evidence-producing systems rather
than answer-producing chatbots. For complex domain datasets, reliability depends
on metric definitions, executable validation, citations, and explicit support
status. The ACA Marketplace benchmark provides a concrete path for evaluating
that claim using real public data and reproducible dbt marts.
