# Claim Boundary

## Claims the project can support

- The project implements a reproducible DuckDB/dbt research substrate over six
  official CMS Plan Year 2026 Marketplace public-use files.
- Direct answering, generated SQL, oracle metric routing, predicted metric
  routing, controlled compilation, validation, execution, and evaluation are
  implemented as distinct stages.
- In this 30-question ACA benchmark, SQL execution success substantially
  overstates strict semantic correctness.
- Under benchmark-specific controlled conditions, the oracle-routed compiler
  produced higher strict and compatible-projection result agreement than the
  subscription Codex-to-SQL pilot.
- Explicit metric definitions, permitted-table records, SQL, validation checks,
  result rows, and citations expose intermediate artifacts for post-hoc failure
  localization and traceability.
- Model-generated paraphrase and challenge diagnostics reveal substantial
  compiler-generalization failures.
- Repeated stochastic outputs are retained and uncertainty is estimated by
  resampling question clusters, not individual repeated calls.

## Claims that require careful qualification

- **“Metric grounding improved result match.”** Safe only as: “In this
  benchmark-specific oracle/predicted-routing experiment, controlled compilation
  had higher result agreement than the evaluated subscription Codex-to-SQL
  condition.” It is not a universal model comparison.
- **“Held-out.”** The close paraphrases are held out from threshold selection but
  derived from training prototypes. The challenge is hash-locked but
  model-generated. Neither is an independently human-authored confirmatory set.
- **“Faithfulness.”** Automated scoring checks numeric evidence presence. Full
  claim-to-result entailment remains pending independent human review.
- **“SQL validity.”** The implemented validator measures static policy
  compliance. DuckDB execution is the separate executable-SQL measure.
- **“Codex.”** Results apply to a logged-in, subscription-backed Codex CLI with
  system instructions and an undisclosed default model snapshot, not a raw API
  model.

## Claims the project cannot support

- General healthcare reasoning ability.
- Generalization to unseen databases, schemas, hospitals, claims data, clinical
  workflows, or other regulated domains.
- Universal LLM hallucination reduction.
- A causal relationship between quality ratings and premiums.
- Enrollment-weighted consumer price, subsidy, utilization, claims, or outcome
  conclusions.
- Learned end-to-end semantic parsing from the oracle compiler.
- 100% held-out accuracy or solved text-to-SQL.
- Independent human validation of the current paraphrase/challenge sets.
- Completed blinded human-review findings.
- Production-ready medical, insurance-selection, or clinical decision support.
- Deployment, peer-reviewed publication, acceptance, or external replication.

## Required public wording

Use **“research prototype”**, **“benchmark-specific”**, **“oracle-routed upper
bound”**, **“model-generated diagnostic set”**, **“subscription Codex pilot”**,
and **“manuscript in preparation.”**

Do not use **“proven,” “general,” “clinical,” “production-ready,” “published,”**
or **“100% held-out accuracy.”**

## Current publication gate

The repository is suitable for a PhD application as an honest work-in-progress
sample. Workshop submission still requires:

1. independent human validation or authorship of a locked confirmatory set;
2. two completed system-label-blinded reviews with agreement/adjudication;
3. a clean tagged rerun with immutable result artifacts;
4. second-reviewer verification of all reference SQL.
