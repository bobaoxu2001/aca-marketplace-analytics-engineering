# Limitations and Next Steps

This project is intentionally honest about what it does and does not prove. It
uses real public CMS data and production-style analytics engineering patterns,
but it is not a deployed production insurance analytics platform.
The v2 mart includes the PY2026 Quality PUF, but not the broader Nationwide QRS
reporting-unit dataset.

## Current limitations

### Public CMS PUFs are not claims-level data

The selected PUFs describe plan design, premium rates, service areas, and benefit
cost sharing. They do not include claims, utilization, provider contracts,
medical cost, or member behavior.

### No member-level PHI is used

The project uses public aggregate/reference files only. There is no PHI,
member-level data, enrollment table, or patient journey modeling.

### Premium metrics are not enrollment weighted

Premium summaries are calculated over Rate PUF rows. They are useful for market
structure and pricing analysis, but they do not represent member-weighted
average premiums.

### Local DuckDB is not a production deployment

DuckDB is used for reproducible local development and validation. Production
deployment would require orchestration, storage, warehouse access controls,
monitoring, CI/CD, and data freshness checks.

### Dashboard is a static preview

`assets/dashboard_preview.png` is generated from real aggregate query outputs.
It is not a deployed BI dashboard, and it does not include interactivity,
permissions, or scheduled refresh.

### County names need enrichment

CMS Service Area `County` values are enriched with a Census FIPS reference seed
(`county_fips_reference`) that adds `county_display_name`, `county_fips`, and
`state_name` to `dim_geography` when a match is found. Unmatched county values
retain the raw CMS value.

### Crosswalk history is one transition, not full SCD history

The current `dim_plan_history` models PY2025 to PY2026 continuity only. A full
SCD implementation would add more plan years, effective date ranges, and
surrogate versioning across repeated annual crosswalk files.

### Quality joins are partially limited by source scope

The plan-level Quality PUF includes some rows that do not join to the modeled
Exchange PUF plan dimension. These rows are retained and flagged with
`joins_to_dim_plan` rather than dropped.

## Recommended next steps

1. **Add Network PUF** to analyze provider network breadth and network design.
2. **Add Nationwide QRS PUF** if reporting-unit/product-level quality measures
   are needed beyond plan-level Quality PUF star ratings.
3. **Extend Plan ID crosswalk / SCD modeling** to additional plan years.
4. **Create a BigQuery or Snowflake profile** for cloud warehouse deployment
   (see `dbt_project/profiles.example.yml`).
5. **Expand CI pipeline** to run dbt build against a cached sample of real CMS
   data on each pull request.
6. **Publish dbt docs locally** with lineage screenshots generated by reviewers,
   not fabricated in the repo.

Implemented in this repository:

- County FIPS reference seed enriching `dim_geography` with display names
- GitHub Actions CI (`dbt parse` + unit tests)
- dbt exposures linking dashboard and LookML to mart models
- Raw load audit freshness tests and `assert_all_raw_tables_loaded`

## dbt lineage preview

To generate lineage locally:

```bash
cd dbt_project
dbt docs generate --profiles-dir .
dbt docs serve --profiles-dir .
```

Then open the local dbt docs site and use the lineage graph to inspect source,
staging, intermediate, and mart dependencies.
